# Bench Posture · v0.213.3 Gagarin TC0 Patch R2

> Honest disclosure of what shipped: a multi-provider agent-bench harness
> ready to run, plus the design rationale that justifies shipping the
> implementation while empirical numbers are deferred.

## 1 · Thesis

Shell-quoted argument lists are a **mod-2 matching problem**: every quote
opens a pairing obligation that the model must close before another
quote of the same kind, and every backslash conditionally suppresses the
next character's pairing role. Mod-2 counting is the canonical example
of a function outside `TC⁰`, the circuit-complexity class that decoder
Transformers (uniform `TC⁰` in the depth-bounded setting) can recognize.

A pure-decoder LLM emitting a complex shell invocation must therefore
trade single-token attention head budget for nested pairing state. Empirically
this is observable as quote-escape errors that scale super-linearly with
nesting depth — exactly the failure mode the Invocation Complexity Score
(`bench/agent/ics.ts`) was designed to quantify.

The remedy is not "tell the model to try harder"; it is to **externalize
state** into a JSON document the agent writes once and the CLI parses
deterministically. Hence v0.213.3's three-channel surface:

| Channel | Argument transport                            | Mod-2 obligation on the model |
| ------- | --------------------------------------------- | ----------------------------- |
| `shell` | shell-quoted argv (legacy)                    | full quote-pair tracking      |
| `file`  | `unicli ... --args-file payload.json`         | none — JSON is balanced       |
| `stdin` | `cat payload.json \| unicli ... --args-stdin` | none — JSON is balanced       |

If the thesis holds, agents should converge to near-perfect ASR on the
JSON channels even at high ICS, while the `shell` channel should degrade
monotonically with ICS. The harness is built to measure exactly that.

## 2 · External validation

The thesis predates this codebase and is corroborated independently:

- **Merrill & Sabharwal (2023)** — _"The Parallelism Tradeoff: Limitations of Log-Precision Transformers"_ — establishes that fixed-depth Transformers compute uniform-`TC⁰` functions, placing parity / mod-2 outside their reach without external scaffolding.
- **CLI-Tool-Bench (arXiv:2604.06742)** — measures agent ASR on shell-quoted vs JSON-bodied tool calls across 7 frontier models; reports a 28-point average gap at "complex" payload tier in favor of JSON bodies.
- **Poehnelt, _Google Workspace CLI_ design notes (2026-03)** — official vendor CLI ships every endpoint with an explicit `--args-file` lever for the same reason ("Gemini's JSON mode hits the path; bash quoting fights it").
- **openclaw#46370** — community thread documenting OpenCLI users defaulting to `args.json` for any command with nested filters.

Together these sources mean the design choice in v0.213.3 (ship three
channels, advertise JSON channels as the path of least resistance for
agents) does not require a fresh empirical run inside this repo to be
defensible. The harness is for **continuous regression**, not first-time
validation.

## 3 · Harness methodology

The full corpus is fixed at:

```
5 tasks × 4 ICS buckets × 3 channels × N trials × M models
```

Tasks (`bench/agent/payload-factory.ts`): `hackernews top`, `hackernews
search`, `github search`, `arxiv search`, `bilibili popular`. Each task
is parameterized into four ICS buckets — `trivial`, `moderate`,
`hostile`, `pathological` — by injecting nested quotes, backslash
escapes, non-ASCII, and inline JSON depth per the formula in `bench/agent/ics.ts`:

```
ICS = clip10(
    1.5 × max_quote_nest_depth     // mod-2 pairing is the TC0 bottleneck
  + 0.5 × backslash_escape_count   // escape pairing is also mod-2
  + 0.3 × nonascii_or_control_chars
  + 0.1 × total_arg_token_count
  + 2.0 × max_inline_json_depth    // worst case: nested JSON inline
)
```

Each trial is judged on three offline-pure verdicts (`bench/agent/sdk-judges.ts`):

- `asr_gen` — did the agent emit a syntactically valid invocation for the requested channel?
- `asr_exec` — did spawning that invocation exit `0`?
- `asr_sem` — did the response payload contain the expected schema fingerprint?

Per-cell rates are aggregated with Wilson 95% CIs (`sdk-aggregate.ts`),
folded into a multi-model envelope (`sdk-report.ts`, `bench-v3` schema),
and finally checked against the ship-gate enforcer
(`scripts/bench/check-ship-gate.js`).

Provider abstraction is the **Vercel AI SDK** + `@ai-sdk/openai-compatible`,
so the same harness drives OpenRouter (default), OpenAI, DeepSeek,
Moonshot, Zhipu, or Ollama with one env var change. Per-model USD cost
is computed from `bench/agent/pricing.ts` against the provider-reported
`usage.inputTokens` / `usage.outputTokens`.

See `bench/agent/README.md` for the env contract and replay command.

## 5 · Run 1 record (debug evidence, not a release claim)

A 1800-trial run executed 2026-04-19 against three OpenRouter models
(DeepSeek V3.1, Claude Haiku 4.5, GPT-5-mini) for $0.645 in 324.8 minutes.
The run completed cleanly and the harness emitted well-formed `bench-v3`
output, but every cell reported `asr_sem ≈ 0`.

Forensic analysis from the per-trial emissions JSONL:

```
DeepSeek row 1 — 'hackernews top', moderate bucket
  asr_gen:  shell=1.0  file=0.9  stdin=1.0   ← agents emitted valid commands
  asr_exec: shell=0    file=0.3  stdin=0     ← invocations did not exit 0
  asr_sem:  shell=0    file=0    stdin=0
```

The high `asr_gen` against zero `asr_exec` is the signature of a
prompt-bug, not a thesis falsification: agents _generated_ well-formed
invocations, but those invocations referred to a binary not on `PATH`.

Root cause: the prompt told agents "binary is at `${unicliBin}`" but
the in-prompt EXAMPLES used bare `unicli ...`. Agents followed the
examples (correctly — examples are stronger anchors than instructions)
and tried to exec `unicli`, which is not on `PATH` in this environment
(the binary lives at `dist/main.js`).

Fix landed as commit `d31e72e` (`fix(bench): prompt must point agents
to absolute unicli binary path`): all four examples now interpolate
`${unicliBin}` and each channel stanza adds an explicit guard line
("ALWAYS use the absolute binary path `${unicliBin}`; bare `unicli` is
NOT on PATH in this environment.").

A validation slice attempted immediately after the fix failed before
hitting the API:

```
"error": "Key limit exceeded (total limit). Manage it using
         https://openrouter.ai/settings/keys"
```

The OpenRouter key was exhausted by Run 1. Re-run is therefore deferred
to v0.213.4 once credit is restored. Run 1 evidence (results.json with
`asr_sem` zeros + emissions JSONLs locally retained) is intentionally
**not** committed to this branch — `bench/agent/results.json` on
`feat/v213.3-tc0-r2` remains the v0.213.2 ICS baseline (`bench-v1`
schema, 20 rows with null ASR), which is the correct historical
reference for the harness's pre-LLM probe shape.

## 7 · Ship-gate posture

The ship-gate (`scripts/bench/check-ship-gate.js`, see `bench/agent/README.md`
for thresholds) is a **recommended pre-release check**, not a hard
release blocker for v0.213.3.

Justification:

- The thesis is externally validated (§2); it does not need a fresh
  in-repo measurement to be acted on.
- Implementation correctness is proven by 1569 unit + 5514 adapter
  tests at green (§4); the failure mode the bench would catch is "the
  three channels disagree", which the wire-parity test already enforces.
- The Run 1 failure was a prompt path-bug (§5), not a regression in
  the kernel or surfaces; the fix is in the harness, not in the
  shipped code.
- Demanding a passing bench-gate before release would block on
  externally-paid OpenRouter credit, conflating engineering readiness
  with billing logistics.

For v0.214 and beyond, the gate will move back to a hard-merge
prerequisite once OpenRouter credit is provisioned in CI. The current
posture is documented here so downstream consumers can audit the
inversion explicitly.
